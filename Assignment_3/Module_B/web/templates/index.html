💥 Atomicity Test

A transaction spanning Assignment 2 relations is crashed mid-way. The database must be identical to the pre-crash state.

What Happens
  1. Transaction begins across Booking, Facility, and Complaint
  2. Creates a temporary booking for member M01
  3. Changes Facility 1 status to Maintenance and files a complaint
  4. ⚡ Power failure simulated — CRASH injected
  5. ROLLBACK undoes all changes using WAL before_images
  6. Pre-state == Post-state → Atomicity verified

✅ Consistency Test

An invalid facility state must be caught before any B+ Tree mutation occurs.

What Happens
  1. Transaction begins
  2. Attempts to set Facility 2 status to Broken (invalid)
  3. Constraint check fires before the WAL write
  4. Transaction rolled back — B+ Tree never touched
  5. Facility state unchanged → Consistency verified

🔒 Isolation Test

Two concurrent transactions run simultaneously. Transaction B must never see Transaction A's uncommitted facility update.

What Happens
  1. Txn A and Txn B both start at the same instant (barrier sync)
  2. Txn A reads Facility 2 status, waits 80ms, then writes Maintenance
  3. Txn B reads Facility 2 status immediately — must see original value
  4. Table-level threading.RLock serialises access
  5. No dirty reads → Isolation verified

🔨 Durability Test

Committed booking data must survive a system restart. The WAL is the only persistence layer.

What Happens
  1. INSERT a new booking and COMMIT (WAL is fsync'd)
  2. Wipe the in-memory B+ Tree (simulates power-off)
  3. Record is gone from memory
  4. ARIES recovery replays WAL: REDO committed txns
  5. Record reappears → Durability verified

♻️ ARIES Crash Recovery

3-phase recovery: Analysis → REDO committed txns → UNDO losers.

How Recovery Works
Phase 1: Analysis Scan WAL. Classify each txn as committed, aborted, or loser (no COMMIT record).
→
Phase 2: REDO Replay all ops for committed txns using after_images. Re-applies lost writes.
→
Phase 3: UNDO Reverse all ops for loser txns in reverse order using before_images.

🏁 Race Condition Test

Identify a critical booking operation, fire many concurrent members at the same facility slot, and verify there is no double booking or corrupted final state.

What Happens
  1. Create one temporary facility slot for booking
  2. Spawn many members trying to reserve that exact same slot together
  3. Hold the booking path under shared transaction locks while reads and writes happen
  4. Verify successes, rejections, and final booking count all match perfectly
  5. Clean up temporary records after the run

⚙️ Stress Testing

Run hundreds or thousands of requests, observe system behaviour under load, and measure correctness plus response time.

What Happens
  1. Launch many concurrent requests using an locust
  2. Mix repeated table snapshots and WAL summary reads
  3. Check every response shape for correctness
  4. Measure throughput, average latency, p95 latency, and max latency
  5. Report all failures seen under load
